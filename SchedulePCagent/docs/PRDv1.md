可以借，但**不能照着做成“另一个 AirJelly”**。
因为 AirJelly 的核心叙事是：

> **“我懂你的工作流，所以我替你推进工作。”**

而你们 **光合日历** 真正有机会赢的，不是“更强监控、更强自动化”，而是：

> **“我懂你的生活节奏，所以我帮你过得更稳、更轻、更有生命感。”**

这两个方向表面都叫 **desktop agent**，但底层产品哲学完全不是一回事。
如果你们抄的是“形”，最后会变成一个弱版办公助手；
如果你们借的是“骨架”，你们反而可能做出一个**很不一样、而且更有用户情感粘性**的产品。

# 一句话先下判断

## 这个项目最值得你借鉴的不是“功能”，而是这 3 个产品结构：

1. **Always-on desktop presence（常驻桌面存在感）**
   用户不需要“打开 App 才开始用 AI”，AI 就在那儿。

2. **Cross-app context → structured memory（跨应用上下文 → 结构化记忆）**
   不是聊天记录堆积，而是把电脑上的碎片信息，整理成“可行动的生活/任务上下文”。

3. **Proactive loop（主动触发循环）**
   不等用户提问，而是在“合适时机”主动出现，推进下一步。

---

但你们必须做一个非常关键的转向：

# 你们不要做「工作代理」，要做「生活节奏代理（Life Rhythm Agent）」

这就是你们的差异化主轴。

# 一、先把 AirJelly 拆开：它到底厉害在哪？

我不看它文案，我只看它产品机制。
它本质上是一个：

# **Context Harness + Agent Runtime + Proactive Surface**

也就是三层：

---

## 1）Harness（采集层）

它不断观察：

* Slack 说了什么
* Zoom 开了什么会
* Docs 写了什么
* Calendar 有什么安排
* 浏览器看了什么

它不是“助手”，它先是一个**持续感知器**。

### 你可以理解成：

> “不是 AI 会不会做事，而是 AI 有没有资格做事。”

没有上下文，Agent 就是瞎子。
Harness 火，是因为大家终于意识到：

> **Agent 的上限，不取决于模型多聪明，而取决于它对真实世界的接入有多深。**

---

## 2）Context Layer（理解层）

它不是简单录屏或日志，而是把观察到的东西转成：

* 人在干什么
* 哪些是任务
* 哪些是承诺
* 哪些东西快过期了
* 哪些工作中断了
* 哪些事情应该接着做

这一步非常重要，因为原始数据本身没有价值，**结构化理解**才有价值。

---

## 3）Proactive Agent（行动层）

然后 AI 才开始主动做事：

* 自动生成待办
* 做 follow-up
* 会后总结
* 明天提醒
* 提前准备 brief
* 补漏

这就是它真正的爽点：

> **“我没说，但它知道。”**

# 二、你们为什么“可以借鉴”，而且很适合做桌面端？

因为你们的产品天然就需要一个 **“后台观察 + 轻提醒 + 低打扰 + 主动陪伴”** 的存在形态。

说白了：

## 光合日历如果只待在手机里，天花板其实有限

因为用户的大量“生活失序瞬间”，并不发生在手机里，而是发生在：

* 电脑工作到发懵的时候
* 开一堆标签页拖延的时候
* 写文档写到卡住的时候
* 临时被消息打断的时候
* 明明知道要做，但迟迟没开始的时候

这些时刻，**桌面端比手机更接近真实问题现场**。

所以你们做桌面端不是“多一个端”，而是：

> **把“理解用户节奏”的能力，真正落到用户最容易失控的场景里。**

这非常对。

# 三、但你们不能学 AirJelly 的“价值叙事”，否则会走偏

AirJelly 的核心用户是：

* Product / Ops / Founder / Researcher
* 高强度电脑工作者
* 追求效率和推进

而你们的核心用户其实是：

* 拖延自责型
* 隐性耗竭型
* 容易失序、内耗、碎片化的人

这意味着一个很大的产品分叉：

---

## AirJelly 的核心问题是：

> **“工作有没有推进？”**

## 光合日历的核心问题应该是：

> **“人有没有被压垮？”**

---

这决定了你们的桌面 agent 不能只是：

* 自动抓任务
* 自动总结
* 自动提醒
* 自动推进

因为这些都容易变成：

> **“更高级的监工”**

而你们原本最宝贵的东西，恰恰是：

* 去监工化
* 去内疚化
* 去绩效化
* 去“我又没做好”的评价体系

所以你们必须守住一句产品原则：

# **任何“主动能力”，都必须服务于“减压”和“恢复节奏”，而不是单纯加码效率。**

这个原则非常重要。
不然你们做着做着，最后就把“光合”做成了“会说话的滴答清单”。

那就完了。

# 四、真正适合你们借鉴的，不是 AirJelly 的“工作代理”，而是它的产品骨架

我给你直接翻译成适合光合日历的版本：

---

# 你们的桌面 Agent 应该叫：

# **生活上下文代理 / 节奏代理 / 陪伴式桌面 AI**

而不是“桌面效率助手”。

# 五、你们可以直接借鉴的 6 个模块（而且很适合你们）

---

# 1）Timeline（时间线）——这个你们非常值得做

AirJelly 的 Timeline 很强，因为它解决了：

> “我今天到底都做了什么？”

这件事对于你们用户尤其重要，因为拖延/耗竭型用户经常有一种感受：

* 一天很累
* 但又觉得自己什么都没做
* 然后开始自责

这其实是一个**感知偏差**问题。

---

## 你们可以把 Timeline 改造成：

# **「今日生命流 / 今日节奏轨迹」**

不是冷冰冰的 activity log，而是：

### 记录今天发生了什么：

* 上午深度专注了 42 分钟
* 下午被消息打断 6 次
* 晚上其实完成了一个一直拖着的小任务
* 今天虽然没高产，但比昨天更稳定
* 你在 15:20 之后明显疲惫了

这一下，价值就不一样了。

---

## 你们不要把它做成：

* Slack at 09:14
* Docs at 10:32
* Calendar at 14:05

这太工具了。

---

## 你们应该做成：

### 今天的节奏像这样：

* **启动慢**：前 1 小时处于切换状态
* **中段进入流**：11:10–12:00 有稳定推进
* **下午被打散**：多个小打断削弱连续性
* **夜间回补**：晚间完成了一个心理负担较高事项

你看，这就是 **“工作记录” → “生活理解”** 的差异。

这很光合。

---

# 2）Task Capture（自动任务捕捉）——非常值得做，但要换叙事

AirJelly 是从对话、会议、消息里自动抓任务。
这个能力你们必须有，因为它太实用了。

但你们的差异化不是“也能抓任务”，而是：

# **你们抓的不只是任务，而是“心理负担点”。**

---

## 例如你们可以自动识别：

### 显性任务

* “明天前交报告”
* “给老师回邮件”
* “周五前订票”

### 隐性负担

* “一直想弄但没开始的事情”
* “反复打开又关掉的页面”
* “连续几天提到但没推进的事项”
* “你其实很抗拒的一件事”

这个就厉害了。

因为真正拖垮人的，很多不是任务本身，而是：

> **“悬而未决”带来的认知负荷。**

---

## 所以你们桌面端不该只是：

* Create task
* Set due date

而应该还能做：

### AI 帮你判断这件事属于哪类：

* 一次性杂务
* 需要深度专注
* 情绪阻力高
* 需要外部协作
* 适合碎片时间推进

这比单纯待办强很多。

---

# 3）Daily Report（日终报告）——这个和你们理念天然契合

AirJelly 的 daily report 是“工作总结”。

你们完全可以把它升维成：

# **「今日生命回顾 / 今日能量结算」**

---

## 你们不要只写：

* 完成 4 个任务
* 2 个会议总结
* 1 个 follow-up

这种写法很容易把用户又拉回绩效焦虑。

---

## 你们应该输出更像这样：

### 今日回顾

* 你今天真正推进了 2 件关键事
* 下午状态下滑后，你仍然保住了最重要的一个节点
* 今天不是“低效”，而是“被切碎”
* 那件你拖了 3 天的小事终于开始了，这是今天最重要的进展

### 明天建议

* 上午先做“阻力最高”的 25 分钟启动
* 下午尽量别安排需要连续思考的任务
* 明天最好预留 30 分钟给恢复，而不是继续硬顶

这就是你们的灵魂。

---

# 4）Proactive Assistance（主动帮助）——你们最应该做，但必须克制

这是最危险也最容易做出差异化的地方。

AirJelly 的主动是：

* 该跟进了
* 该准备了
* 该总结了

你们可以做，但要从“推进工作”改成：

# **“在用户最容易掉下去的时候，轻轻扶一把。”**

---

## 你们最适合做的主动触发，不是“更多提醒”，而是 4 类：

---

## A. 启动型触发（帮开始）

适合拖延用户。

比如：

* 你已经围着这件事转了 18 分钟，要不要我帮你切成第一步？
* 这个任务不像“做完型”，更像“打开型”，先开个头就行
* 你现在不是没时间，你是没入口，我来给你一个 10 分钟入口

这很强。

---

## B. 过载型触发（帮减压）

适合隐性耗竭用户。

比如：

* 你今天已经切换了 11 次上下文，不适合再开新任务
* 现在更适合清理型任务，而不是高认知任务
* 你今天承接了很多别人的需求，要不要先把自己的 1 件事保下来

这是真正“陪伴”的 AI，不是催命 AI。

---

## C. 失联型触发（帮捡回来）

比如：

* 你上周一直挂心的那件事，这两天都没碰，要不要我帮你重启
* 这不是“你没坚持”，只是这个任务需要重新入口
* 你是不是忘了这件事？我把上下文给你接回来

这就是 Agent 的真正价值：

> **不是提醒，而是“帮你重新接上自己”。**

---

## D. 节奏型触发（帮配速）

比如：

* 你通常上午更适合做输出型任务
* 这件事放在你低能量时段大概率会卡住
* 今天剩下的精力，更适合做收尾而不是开新坑

这就把“日历”升级成“节奏操作系统”了。

---

# 5）Cross-App Context（跨应用上下文）——你们一定要做，但要非常注意边界

这是桌面 Agent 的命门，也是风险点。

---

## 你们适合接入的上下文，不需要一开始就“全监控”

很多团队一上来就想：

* 截屏
* 全局 OCR
* 所有 app 监听
* 全盘记忆
* 全自动代理

这很酷，但也很容易死。

因为：

1. 技术复杂度高
2. 隐私压力大
3. 用户会害怕
4. 你们会很难先跑通价值闭环

---

## 你们更适合做的是：

# **“轻量、有边界、可控授权”的上下文采集**

比如先只做这几类：

### 第一阶段最值得接：

* 系统日历
* 浏览器标签页标题
* 剪贴板片段
* 用户主动拖入/分享内容
* 通知摘要
* 本地文档/笔记的用户授权读取
* Flutter App 内已有对话和任务系统

---

## 你们不要先做：

* 全量屏幕录制
* 无差别键盘监听
* 所有内容默认永久记忆

这很容易把“陪伴”做成“监视”。

而你们品牌最怕这个。

---

# 6）Desktop Presence（桌面存在感）——这个是你们最大的体验机会

这是很多人忽略的。

桌面 Agent 最重要的，不是功能页，而是：

# **它“平时怎么存在”。**

AirJelly 的方向是“always-on assistant”。
你们更适合做成：

# **低存在感、但有情绪温度的桌面陪伴体**

这和你们“五个精灵伙伴”设定简直天然适配。

---

## 你们桌面端最适合的不是一个“大而全主界面”

而是一个：

### **悬浮的小生命体 / 轻侧边栏 / 系统托盘精灵**

平时很安静，只在合适时机出现：

* 轻轻冒一句
* 帮你接住一下
* 给你一个最小行动建议
* 或者默默整理完等你来看

这非常有辨识度。

---

# 六、真正属于你们的差异化，不是“更会做事”，而是“更懂人会卡在哪”

我直接给你一个定位对照表，你会更清楚：

---

# AirJelly vs 光合桌面 Agent（建议版）

| 维度       | AirJelly    | 光合桌面 Agent 应该做 |
| -------- | ----------- | -------------- |
| 核心目标     | 推进工作        | 稳住生活节奏         |
| 用户感受     | “它帮我省事”     | “它懂我现在为什么做不动”  |
| 数据来源     | 工作流上下文      | 生活/工作混合节奏上下文   |
| 任务处理     | 自动抓任务       | 自动抓任务 + 识别心理阻力 |
| 时间线      | 活动记录        | 生命流 / 节奏轨迹     |
| 主动提醒     | 跟进、总结、准备    | 启动、减压、捡回、配速    |
| 日报       | 工作 recap    | 能量/节奏/成长回顾     |
| Agent 角色 | 执行型副手       | 陪伴型节奏教练        |
| 品牌气质     | 强工作、强 agent | 温柔但不弱、懂状态的 AI  |

# 七、你们最有机会做出的“爆点”，不是全能 Agent，而是 3 个极强场景

不要一开始就做“全自动操作电脑的超级代理”，那很容易变 demo 产品。
你们更应该做的是：

# **先打穿 3 个高频高痛场景。**

---

# 场景 1：拖延启动场景（你们最适合）

### 用户状态：

* 开着电脑
* 打开了一堆东西
* 就是不开始

### 你们可以做：

* 识别“徘徊态”
* 自动总结“你现在其实想做的是这件事”
* 给出一个 5~10 分钟启动动作
* 一键进入“陪跑模式”

### 这就是你们的杀手功能之一：

# **AI 启动陪跑**

比如：

> “你已经在这件事周围转了 17 分钟，要不要先只做第一段？”

这个比“请专注”强太多。

---

# 场景 2：碎片化工作失序场景（隐性耗竭型杀手场景）

### 用户状态：

* 消息很多
* 标签很多
* 任务很多
* 一天下来很忙但没有主线

### 你们可以做：

* 自动抓散落任务
* 合并同主题事项
* 给出“今天其实只剩 1 条主线”
* 帮用户把碎片重组成一个可执行节奏

### 核心价值：

# **AI 帮你把“散”重新收回来**

---

# 场景 3：日终自责场景（极强情感价值）

### 用户状态：

* 晚上打开 App
* 感觉今天又废了

### 你们可以做：

* 自动回顾今天真实推进
* 把不可见的努力显化
* 给出“今天不是失败，而是被打断/耗竭/过载”的解释
* 帮用户保留一点“我并不是毫无进展”的感受

这个非常非常符合你们理念，而且会形成很强留存。

因为很多用户不是缺工具，而是缺：

> **“有人帮我把这一天看明白。”**

这就是产品壁垒。

---

# 八、如果你们真要做桌面端，我建议你们产品路线别从“超级 Agent”起步，而是走这条路线

---

# Phase 1：做「桌面陪伴壳 + 上下文输入 + 主动提醒」

### 目标：

先证明“桌面端存在感”有价值。

### 先做：

* 桌面悬浮精灵 / 托盘助手
* 和 Flutter App 任务/对话同步
* 用户可一键把网页/文本/文件丢给光合
* 自动生成“待处理事项”
* 今日节奏概览
* 日终回顾

### 这一阶段不要急着做：

* 自动操作电脑
* 全局 OS 代理
* 复杂多步 action execution

先证明“它常驻在电脑上，真的更懂我了”。

---

# Phase 2：做「节奏理解引擎」

### 加强：

* 自动识别拖延/切换/过载状态
* 给任务打“阻力标签”
* 给用户打“今日节奏画像”
* 让 AI 不只是安排任务，而是安排**适合当下状态的任务**

这一步会把你们从“AI 待办”拉开。

---

# Phase 3：做「轻量主动代理」

### 再做：

* 自动草拟 follow-up
* 自动整理会议/课堂/资料到某个任务下
* 自动生成第二天计划草案
* 自动把“今天散落的东西”汇总成一个“可继续的入口”

这时候 Agent 才开始真正有味道。

你们不该这么讲。

* **“懂你的节奏，而不只是懂你的任务”**

这就是品牌护城河。
