# 光合日历（Flutter）产品分析 & 桌面扩展定位

> **文档目的**：基于仓库内 `schedule_app_flutter` 的**实际实现**，归纳当前移动端产品的功能与理念；并说明从该基础扩展到 **SchedulePCagent 桌面端** 时，**继承什么、刻意做成什么不同**，与 [PRDv1.md](./PRDv1.md) 对齐、互补。  
> **分析基准**：代码与目录结构（截至文档编写时的仓库状态），不替代正式对外 PRD。

---

## 1. 代码库位置与技术栈

| 项目 | 路径 | 说明 |
|------|------|------|
| Flutter 客户端 | `schedule_app_flutter/` | 主产品壳，与 iOS 行为对齐（注释中大量引用 iOS 控制器命名） |
| 后端 API | `schedule_backend/`（Flutter 通过 `ApiService` 调用） | AI 对话等能力走 HTTP，如 `/ai/chat` |

**Flutter 依赖要点**（`pubspec.yaml`）：`provider` 状态管理、`http` 网络、`shared_preferences` 本地键值、`intl` 日期本地化。  
**入口流程**（`lib/main.dart`）：调试模式可强制从引导重来；正式流程为 **Onboarding V1/V2 → 基本信息流 → 登录注册 → `MainContainerView` 主界面**。

## 2. 产品理念（从实现反推）

当前 Flutter 实现体现的理念与「光合日历」叙事一致，核心是 **生活维度平衡 + 情感化陪伴 + 可视化成长**，而不是纯效率/KPI 工具。

### 2.1 五维生活与「精灵」人格

- **五类精灵**（`lib/models/spirit_type.dart`）：光（工作学习）、水（娱乐休闲）、土壤（健康）、空气（社交）、营养（爱好）。  
- 任务分类、植物评分、许愿均绑定到 **同一套 Spirit 维度**，把「安排」从单线待办变成 **多面向生活结构**。
- **AI 人格**（`lib/services/spirit_prompts.dart`）：群聊为五精灵集合体；私聊按精灵有独立 system prompt（与 iOS 对齐），强调温柔、陪伴、反差萌，**不是冷冰冰的助理口吻**。

### 2.2 「状态植物 / 生命感」而非纯列表

- **植物页**（`lib/pages/plant_page.dart`）：植物状态卡片、**雷达图**（五维分数）、**周报**、**月果实**等，把行为结果映射为 **可看的生长与结果**，贴近「生命树 / 生命可视化」方向。
- 植物状态模型（`lib/models/plant_status.dart`）按周聚合 **各精灵维度分数**，是产品「成长可见」的载体。

### 2.3 引导用户心理与目标叙事

- **Onboarding V1**：多题选择（`onboarding_v1_data.dart` 等），建立初始画像。  
- **Onboarding V2**：关键词方向——**突破 / 修复 / 探索 / 稳定**（`onboarding_flow_page.dart`），直接对应不同心理诉求（进取、疗愈、好奇、平和），**不是只问职业身份**。

### 2.4 AI 与日程的耦合方式

- **日历页浮层** `AIChatOverlay`：在「我的安排」场景下常驻对话入口；支持 **群聊 / 私聊**、精灵选择。  
- **`ChatProvider`**：对接 `AIChatRepository`（当前主路径为 `RemoteAIChatRepository`）；若后端返回 **可解析的创建任务**，可写入 **`TaskProvider`**，实现「对话 → 落到日历」的闭环。

**理念归纳**：移动端已经把 **对话、多维任务、成长可视化、精灵陪伴** 串在同一产品里；缺的是 **电脑工作场景下的上下文与轻打扰陪伴**（这正是桌面 Agent 文档要补的）。

## 3. 功能结构（模块级）

### 3.1 主壳 `MainContainerView`

- 三 Tab 式导航：**我的安排（日历）**、**植物状态**、**我的**。  
- **仅在日历模式**展示：左侧 **三叶草** 切换日/周/月视图、**AI 对话浮层**。  
- 全屏自然风背景图 + 自定义字体，**强氛围、弱「办公软件」感**。

### 3.2 日历 `CalendarPage`

- **日 / 周 / 月** 视图（`day_calendar_view` / `week_calendar_view` / `month_calendar_view`）。  
- 按日展示任务列表（`TaskItem`），任务带 **精灵分类** 等。  
- 入口：**添加任务**、**许愿瓶**（`wish_bottle_page`）等。  
- AI 聊天展开时会 **挤压/调整布局**（`aiChatExpandedNotifier`），保证「对话与日程同屏」体验。

### 3.3 植物 `PlantPage`

- 周切换、植物状态卡、PageView：**月果实 / 雷达 / 周报** 等（与 `PlantProvider`、`LocalPlantRepository` 等配合）。

### 3.4 个人 `ProfilePage`

- 用户信息、功能入口、菜单（实现为产品壳；部分数据仍为占位展示，以代码为准）。

### 3.5 任务与愿望

- **任务** `Task`：标题、描述、起止时间、`SpiritType`、重复选项、是否全天（`lib/models/task.dart`）；持久化经 `TaskProvider` / `LocalTaskRepository` 等。  
- **愿望** `Wish`：标题、内容、精灵、完成态（`lib/models/wish.dart`），与许愿瓶流程相关。

### 3.6 账号与引导

- **登录注册**多页流程（手机号等）。  
- **基本信息流** `BasicInfoFlowPage`（昵称、身份状态等）。  
- 与 `main.dart` 中 `SharedPreferences` 键配合，控制是否进入主界面。

### 3.7 AI 对话

- 远程：`RemoteAIChatRepository` + `ApiService`（可配置 `overrideBaseUrl` 便于真机连本机后端）。  
- 消息模型、发送节流、与任务创建联动见 `ChatProvider` 与 repository 实现。

## 4. 数据流与扩展锚点（桌面可对接处）

| 能力 | 移动端现状 | 桌面 Agent 可承接方式 |
|------|------------|------------------------|
| 对话与任务 | 后端 AI 返回可带 `createdTasks` → 写入本地任务 | 同一用户账号下 **会话 / 任务同步协议**（需后端与 Agent 约定） |
| 精灵维度 | 任务、愿望、植物评分均用 `SpiritType` | 桌面侧摘要、日报、节奏建议 **沿用五维语言**，避免变成「纯办公助手」 |
| 本地优先 | Task/Wish/Plant 大量本地仓库 | 桌面可增加 **节奏信号**（见 PRD），与手机 **合并叙事**（如每日回顾） |

## 5. 扩展到桌面：一样的是什么，不一样的是什么

### 5.1 应继承的「产品基因」（与 PRDv1 一致）

- **节奏与陪伴**：主动能力服务于减压、启动、捡回，而非监工式 KPI（详见 [PRDv1.md](./PRDv1.md) 原则）。  
- **五精灵与植物隐喻**：桌面 UI 宜 **低存在感**（托盘/悬浮精灵），与移动端「自然、温柔」一致。  
- **结构化记忆**：不是聊天记录堆砌，而是 **可行动的日程与回顾**（Timeline / Daily Report 的「生命流」版）。

### 5.2 刻意「不同质」于移动端的部分

| 维度 | 移动端（当前 Flutter） | 桌面 Agent（目标差异） |
|------|------------------------|-------------------------|
| 使用场景 | 随身查看、碎片记录、情绪陪伴 | **电脑前**失序、拖延、碎片化打断、日终自责 |
| 上下文来源 | 主要是用户主动输入 + 日历内数据 | **可选、分阶段**增加系统日历、剪贴板、浏览器标题等（PRD 中的轻量 harness） |
| 主界面 | 全屏三 Tab + 日历为主 | **非全屏办公套件**：侧边条/托盘 + 与手机 **对话延续** 即可 |
| 成功指标 | 完成任务 + 植物成长感 | **稳住节奏、降低内疚叙事**（同一数据，不同解释框架） |

### 5.3 与「办公类桌面 Agent」的区隔（简要）

- 市面常见叙事：*Knows your work / Acts before you ask*。  
- 光合桌面应坚持 PRDv1 的定位：**懂人为何做不动、帮生活节收回笼**，功能形态可借鉴 Timeline/Task Capture，**价值叙事不抄办公赛道**。
